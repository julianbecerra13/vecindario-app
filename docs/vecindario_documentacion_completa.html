1. Vision General del Proyecto

Vecindario es una aplicacion movil para conjuntos residenciales en Colombia. Conecta residentes, administradores, tenderos y proveedores de servicios dentro de una misma comunidad.

Problema que resuelve

  • Comunicacion fragmentada en conjuntos (WhatsApp, carteleras, porteria)
  • Falta de visibilidad de servicios y comercios locales entre vecinos
  • Administracion manual de multas, PQRS, finanzas y asambleas
  • Dificultad para gestionar reservas de zonas sociales

Propuesta de valor

  • Para residentes: Un solo lugar para enterarse de todo, pedir comida al vecino, reservar el salon comunal, pagar la cuota y votar en asamblea
  • Para administradores: Gestion digital completa del conjunto con tracking, SLA y reportes
  • Para tenderos/vecinos: Canal de venta directa dentro de su comunidad

3. Arquitectura y Patrones

Patron principal: Feature-First + Repository Pattern

Cada feature es un modulo autocontenido con sus propias pantallas, modelos, providers y repositorios.

Screen (UI) → Provider (Estado) → Repository (Datos) → Firebase

Flujo de datos

  • Screen: Widget de Flutter que muestra la UI. Usa ConsumerWidget o ConsumerStatefulWidget de Riverpod.
  • Provider: Expone datos reactivos. Usa StreamProvider para datos en tiempo real de Firestore y StateNotifierProvider para estado local mutable.
  • Repository: Capa de abstraccion sobre Firebase. Traduce entre modelos Dart y documentos Firestore.
  • Firebase: Backend as a Service. Firestore para datos, Storage para archivos, Auth para identidad.

Convenciones de codigo

ElementoConvencionEjemplo
Archivossnake_casepost_model.dart
ClasesPascalCasePostModel
ProviderscamelCase + sufijo ProviderfeedPostsProvider
RepositoriosPascalCase + sufijo RepositoryFeedRepository
ModelosPascalCase + sufijo ModelUserModel
ImportsAbsolutospackage:vecindario_app/...

Inyeccion de dependencias

Todo se inyecta via Riverpod. Los repositorios se crean como Provider que reciben instancias de Firebase de otros providers:

final firestoreProvider = Provider((_) => FirebaseFirestore.instance);
final feedRepositoryProvider = Provider((ref) =>
  FeedRepository(ref.watch(firestoreProvider)));
final feedPostsProvider = StreamProvider((ref) =>
  ref.watch(feedRepositoryProvider).watchPosts(communityId));

5. Sistema de Roles y Permisos

La app maneja 4 roles distintos, cada uno con diferentes niveles de acceso:

RolEnumAcceso
ResidenteUserRole.residentFeed, servicios, tiendas, perfil. En premium: leer circulares, ver multas propias, crear PQRS, reservar zonas, ver estado de cuenta, votar en asambleas
AdministradorUserRole.adminTodo lo del residente + panel admin, aprobacion de residentes, fijar posts. En premium: crear circulares con tracking, gestionar multas, responder PQRS, dashboard financiero, convocar asambleas
TenderoUserRole.storeOwnerTodo lo del residente + gestionar su tienda, ver pedidos entrantes, actualizar estado de pedidos
ExternoUserRole.external_Perfil visible en directorio de servicios externos

Flujo de verificacion

Registro → Codigo invitacion → Pantalla espera → Admin aprueba → Acceso completo

Providers de rol

  • isAdminProvider — true si user.role == admin
  • isVerifiedProvider — true si el admin aprobo al residente
  • isPremiumProvider — true si la comunidad tiene suscripcion activa
Capa dual en Premium: Cada pantalla del modulo premium detecta isAdmin y renderiza la vista correspondiente. El admin ve herramientas de gestion; el residente ve la vista de consumo.

Flujo completo de ingreso

  1. Onboarding: 3 pantallas de bienvenida (carrusel) la primera vez
  2. Login/Registro: Email+password o Google Sign-In
  3. Sin comunidad: Redirige a /join-community donde ingresa codigo de 6 caracteres
  4. Ingresa torre y apartamento: Se asocia a la comunidad
  5. Sin verificar: Ve pantalla de espera hasta que un admin lo apruebe
  6. Verificado: Acceso completo al feed y demas

Operaciones de AuthRepository

  • signInWithEmail(email, password) — Login con Firebase Auth
  • signUpWithEmail(email, password, displayName) — Crea cuenta + documento en Firestore
  • signInWithGoogle() — OAuth con Google
  • resetPassword(email) — Envia email de recuperacion
  • signOut() — Cierra sesion

Guards del router

El router tiene redirects automaticos basados en el estado del usuario:

  • No autenticado → /login
  • Sin comunidad → /join-community
  • No verificado → /pending-approval
  • Todo OK → /feed

  • Votar en tiempo real (botones por opcion)
  • Ver resultados con barra de progreso y %

Modelo: AssemblyModel

  • status — convened (convocada), active (en vivo), closed (cerrada)
  • votes: List<VoteItem> — Lista de votaciones
  • attendees — Lista de UIDs presentes
  • actaPdfURL — Acta en PDF una vez cerrada

Modelo: VoteItem

  • topic — Tema de la votacion
  • options — Opciones (ej: "A favor", "En contra", "Abstencion")
  • results — Mapa opcion → lista de UIDs que votaron
  • hasVoted(uid) — Previene doble voto
  • percentageFor(option) — Calcula % de cada opcion

Votacion en tiempo real

Usa FirebaseFirestore.runTransaction() para garantizar atomicidad. Cuando un residente vota, los demas ven los resultados actualizarse instantaneamente via Firestore streams.

26. Cumplimiento Legal - Ley 1581/2012

Vecindario cumple con la Ley de Proteccion de Datos Personales de Colombia (Habeas Data):

Derechos implementados

DerechoImplementacion
ConocerPantalla de privacidad explica que datos se recopilan
ActualizarEditar perfil permite modificar nombre, telefono, foto
RectificarEl usuario puede corregir sus datos en cualquier momento
SuprimirBoton "Eliminar mi cuenta" en pantalla de privacidad
RevocarEl usuario puede revocar consentimiento y cerrar cuenta
PortabilidadBoton "Exportar mis datos" genera archivo con toda la info

Flujo de eliminacion

Usuario solicita → Dialogo de confirmacion → Se crea deletionRequest → Cloud Function anonimiza datos → Se desloguea

Colecciones de compliance

  • deletionRequests/ — Solicitudes de eliminacion pendientes
  • consents/ — Registros de consentimiento
  • auditLogs/ — Logs de acceso a datos sensibles
Soft delete: Los datos no se borran fisicamente sino que se anonimizan (displayName → "Usuario eliminado", email → deleted_{timestamp}@...) para mantener la integridad de posts, resenas y pedidos historicos.
